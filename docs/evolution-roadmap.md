# خارطة طريق التطوّر — Hub مركزي ومرن

هذا المستند يصف الاتجاه الجديد للمنصة بعد إكمال Phase 00–06: **استضافة الـ Hub على سيرفر** مع **بانل إدارة** لتعديل الأدوات والمهارات والإضافات دون نشر كود، وإمكانية **أدوار (Roles)** لتحكم من يرى ماذا (مفيد لـ Cursor وغيره).

---

## الأهداف الرئيسية

| الهدف | الوصف |
|-------|--------|
| **استضافة على سيرفر** | تشغيل الـ Hub كخدمة شبكية (HTTP/SSE) بدل stdio فقط؛ المبرمجون يتصلون برابط واحد. |
| **بانل إدارة** | واجهة (ويب) لإضافة/تعديل/حذف Tools و Skills و External MCP Plugins دون الحاجة لتعديل الكود أو إعادة النشر. |
| **مرونة التغيير** | أي تغيير في القدرات (أدوات، مهارات، إضافات) يُدار من البانل ويُنعكس فوراً على كل المتصلين. |
| **أدوار (Roles) — اختياري** | ربط الأدوات/المهارات/الإضافات بأدوار؛ Cursor (أو أي عميل) يمرّر دور المستخدم فيصل فقط قائمة الأدوات المسموح بها له، فيقل التشويش ويزيد الأمان. |

---

## كيف يفيد Cursor من الأدوار؟

- **بدون أدوار:** Cursor يستدعي `tools/list` فيحصل على كل الأدوات؛ الـ AI يختار من بينها. قد يكون العدد كبيراً أو تحتوي أدوات حساسة.
- **مع أدوار:** العميل يرسل مع الاتصال (أو مع طلب initialize) **معرف دور** (مثلاً `developer`, `admin`). الـ Hub يفلتر `tools/list` حسب الصلاحيات المرتبطة بذلك الدور فيرجع فقط الأدوات المسموح بها. النتيجة:
  - واجهة أوضح للـ AI (أدوات أقل وأكثر صلة).
  - تحكم أمان: أدوات خطرة أو إدارية تظهر فقط لـ `admin` أو أدوار محددة.

## ملاحظات

- يمكن تنفيذ **07** ثم **08** ثم **09** بالترتيب، أو تنفيذ 07 ثم 09 (أدوار بدون بانل، بإعداد ثابت للأدوار).
- البانل (08) يمكن أن يكون تطبيق ويب منفصل يتصل بـ API أو قاعدة بيانات مشتركة مع الـ Hub.
- الأدوار (09) تحتاج اتفاقية بين الـ Hub والعميل (Cursor) لتمرير معرف الدور (مثلاً header أو حقل في initialize).
