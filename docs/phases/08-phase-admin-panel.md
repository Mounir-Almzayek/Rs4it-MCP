# Phase 08 — بانل الإدارة (Admin Panel)

## الهدف

توفير واجهة إدارة (ويب) تسمح بإضافة وتعديل وحذف **Tools** و **Skills** و **External MCP Plugins** دون الحاجة لتعديل الكود أو إعادة نشر الـ Hub، بحيث يصبح التغيير مرناً ومتاحاً من مكان واحد.

---

## المخرجات المتوقعة

- تطبيق ويب (بانل) يمكن لمدراء النظام أو المطوّرين الدخول إليه.
- من البانل: إدارة **أدوات** (إضافة أداة جديدة بوصف ومخطط مدخلات وربط تنفيذ، أو تفعيل/تعطيل أدوات موجودة).
- من البانل: إدارة **مهارات** (إضافة مهارة جديدة بوصف ومخطط وربط handler أو قالب، أو تفعيل/تعطيل).
- من البانل: إدارة **إضافات MCP خارجية** (إضافة/تعديل/حذف إدخالات من نوع config الإضافات: id, command, args، إلخ).
- التغييرات تُخزَن في تكوين ديناميكي (قاعدة بيانات أو ملفات إعداد يُحدَّث) والـ Hub يقرأها ويُحدّث قائمة الأدوات دون إعادة تشغيل (أو بإعادة تحميل تكوين عند الطلب).

---

## المهام الفرعية

### 8.1 نموذج التخزين للتكوين الديناميكي

- [ ] تحديد أين يُخزَن التكوين: قاعدة بيانات (SQLite، PostgreSQL، إلخ) أو ملفات JSON يُحدَّثها البانل.
- [ ] تعريف نموذج بيانات للأدوات والمهارات والإضافات القابلة للإدارة (حقول مشتركة مع الـ Hub الحالي).
- [ ] واجهة أو طبقة (مثلاً `src/admin/store.ts`) لقراءة/كتابة هذا التكوين.

### 8.2 API للبانل

- [ ] نقاط نهاية (REST أو غيره) للبانل: قراءة قوائم الأدوات/المهارات/الإضافات، إضافة، تعديل، حذف.
- [ ] حماية هذه النقاط (مصادقة/صلاحيات) حتى لا يصل لها إلا مدراء أو مستخدمون مفوضون.
- [ ] عند التعديل: إما إشعار الـ Hub بإعادة تحميل التكوين، أو الـ Hub يقرأ من المصدر المشترك عند كل `tools/list` (حسب التصميم).

### 8.3 ربط الـ Hub بالتكوين الديناميكي

- [ ] الـ Hub يقرأ قائمة الأدوات والمهارات والإضافات من مصدر التكوين (إضافةً إلى أو بدل الأدوات/المهارات المضمّنة في الكود).
- [ ] آلية تحميل: عند بدء التشغيل و/أو عند إشعار من البانل (webhook أو polling) لإعادة تحميل التكوين.
- [ ] أدوات/مهارات "ديناميكية": تنفيذها إما عبر كود قوالب (مثلاً مهارة تعتمد على سلسلة استدعاء أدوات مُعرّفة في التكوين) أو عبر ربط handler مُسجّل مسبقاً بالاسم.

### 8.4 واجهة المستخدم (البانل)

- [ ] شاشات لـ: قائمة الأدوات (مع إضافة/تعديل/حذف)، قائمة المهارات، قائمة الإضافات الخارجية.
- [ ] نماذج إدخال مناسبة (اسم، وصف، مخطط مدخلات، أمر تشغيل للإضافات، إلخ).
- [ ] (اختياري) معاينة سريعة لـ `tools/list` كما يراها العميل.

### 8.5 التوثيق والأمان

- [ ] توثيق كيفية الوصول للبانل وإدارة الصلاحيات.
- [ ] عدم تعريض API البانل أو واجهته للإنترنت بدون مصادقة مناسبة.

---

## معايير الإكمال

- من البانل يمكن إضافة/تعديل/حذف إدخال أدوات ومهارات وإضافات.
- الـ Hub يعكس هذه التغييرات (خلال مدة مقبولة) ويُعيد `tools/list` محدّثة.
- استدعاء أداة/مهارة/إضافة مُضافة من البانل يعمل بنجاح.

---

## التبعيات

- **Phase 07** مُفضّل (استضافة الـ Hub على سيرفر) حتى يكون البانل والـ Hub على بيئة مشتركة؛ يمكن نظرياً تشغيل البانل محلياً ضد Hub محلي للاختبار.

---

## الملفات المقترحة

| الملف / المجلد | الغرض |
|----------------|--------|
| `src/admin/` أو مشروع منفصل `admin/` | منطق تخزين التكوين و/أو API البانل |
| `src/config/dynamic-config.ts` (أو مشابه) | قراءة تكوين الأدوات/المهارات/الإضافات من المصدر الديناميكي |
| واجهة البانل | مشروع ويب (React، Vue، إلخ) أو صفحات بسيطة تتصل بـ API البانل |

---

## ملاحظات

- يمكن بناء البانل كجزء من نفس الـ repo أو كتطبيق منفصل يتصل بـ API الـ Hub أو بقاعدة مشتركة.
- الأدوات/المهارات المُضافة من البانل قد تكون "قوالب" (تسلسل أدوات موجودة) في البداية؛ تنفيذ كود عشوائي من البانل يتطلب آليات أمان إضافية.
